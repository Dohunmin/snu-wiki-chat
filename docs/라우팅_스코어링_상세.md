# 라우팅·스코어링·LLM 입력 — 실제 예시로 이해하기

> **작성일**: 2026-05-08
> **대상**: SNU 거버넌스 위키 챗 (8 wikis + 1 lens persona)
> **목적**: "이 질문이 들어오면 LLM이 정확히 무엇을 보는가"를 실제 데이터로 추적

---

## 0. TL;DR (가장 중요한 5가지)

1. **LLM은 source의 청크 + stance/fact/overview 통째 + 매칭된 entity 본문만 읽음**
2. **Topic 본문은 LLM이 못 봄.** Topic은 "관련 source를 끌어오는 다리"일 뿐
3. 라우팅은 **3단계** — globalKeyword → keyword 매칭 → preScore + concept-index
4. 위키별로 청크 cap이 분배됨 (현재 30 chunks ÷ 6 wikis ≈ 5/wiki)
5. **history 위키는 메타가 빈약(topics·entities 0)**해서 라우팅 누락 위험 있음

---

## 1. 흐름 도식

```
사용자 질문
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Stage A: routeQuery() — 어느 위키들을 쓸까?      │
│  Tier 0: globalKeyword? → 8개 위키 모두          │
│  Tier 1: keyword 매칭 + preScore                 │
│  Tier 2: concept-index → forced wikis             │
│  → 최대 6개 위키 선정                              │
└──────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Stage B: 각 위키 getContext()                     │
│  - source 점수 (topics·entities·tags·content)    │
│  - source 청크 분할 (## 헤더 단위)                │
│  - 청크별 점수 (단어 빈도 + guaranteed 부스트)    │
│  - stance/fact/overview 통째 점수                 │
│  - 위키별 chunkCap만큼 잘라냄                     │
└──────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Stage C: LLM 입력 조립                             │
│  system prompt + user message (위키별 자료 블록)  │
└──────────────────────────────────────────────────┘
    │
    ▼
Claude 호출
```

## 📖 예시 1 — "시흥캠퍼스 배곧병원 진행 상황은?"

### Stage A 추적

**쿼리 단어**: `시흥캠퍼스`, `배곧병원`, `진행`, `상황`

**각 위키 prefilterScore**:

| 위키 | 매칭된 키워드 | 점수 | 비고 |
|---|---|---:|---|
| senate | "시흥-캠퍼스", "배곧병원", "시흥캠퍼스" | ~12 | preScore 매칭(+5) |
| board | "시흥캠퍼스", "배곧서울대학교병원" | ~10 | preScore 매칭(+5) |
| vision | "시흥캠퍼스", "멀티캠퍼스" | ~8 | |
| plan | "시흥캠퍼스" | ~5 | |
| status | — | 0 | **alwaysContext로 강제 포함** |
| history | — | 0 | 키워드 매칭 0 |
| yhl-speeches | — | 0 | |
| finance | — | 0 | |

**선택된 위키**: `senate`, `board`, `vision`, `plan`, `status` (5개)

### Stage B — senate 위키 내부

**관련 source 후보**:

| Source ID | Title | source.topics | 점수 |
|---|---|---|---:|
| `17기-11차` | 제17기 평의원회 제11차 본회의 요약 | `시흥 캠퍼스`, `평의원회 정책연구과제` | +3 (topics) +5 (content "시흥") = ~8 |
| `17기-8차` | 제17기 평의원회 제8차 본회의 요약 | `ERP 구축`, `시흥 캠퍼스` | +3 +3 = ~6 |
| `17기-12차` | 제17기 평의원회 제12차 본회의 요약 | (시흥 직접 매칭 X, 캠퍼스·인프라) | +1 |

`data.topics`에 `시흥-캠퍼스` topic이 있으면 → 그 topic의 `sources` 배열에 있는 ID들이 `guaranteedIds`에 추가 → **+5 추가 부스트**

**청크 분할 후**:
- `17기-11차` 본문이 `## ` 헤더 5개로 분할 → 청크 5개
- 각 청크 점수 = 쿼리 단어 등장 횟수
- guaranteed 청크는 점수 × 2 + 1

**위키별 chunkCap**: 5개 위키 (status 1 + 일반 4) → status 5청크 + 일반 ~6청크씩

**최종 senate 선택**:
- 17기-11차 첫 청크 (시흥 직접 언급 부분)
- 17기-8차 첫 청크
- 17기-11차 둘째 청크 (점수 높은 게 남으면)
- ...

### Stage C — LLM 입력 미리보기

```
## 위키 자료

### [평의원회] 관련 자료

## 제17기 평의원회 제11차 본회의 요약 (17기-11차) | 회의일: 2022-10-27
{시흥 캠퍼스 정책연구 결과 보고 청크}

---

## 제17기 평의원회 제8차 본회의 요약 (17기-8차) | 회의일: 2022-06-16
{시흥 캠퍼스 ERP 관련 청크}

---

### [이사회] 관련 자료

## 2024년 제4차 이사회 의사록 요약 (2024-4차)
{배곧병원 신축 의결 청크}

---

(vision, plan, status 위키 자료 이어짐...)

---

## 질문

시흥캠퍼스 배곧병원 진행 상황은?
```

LLM은 위 자료만 가지고 답변. **`시흥-캠퍼스` topic 본문 자체는 LLM이 못 봄** — topic은 senate의 어떤 source를 가져올지 결정하는 다리 역할만 함.

## 📖 예시 2 — "법인화 이후 서울대 재정 변화"

### Stage A 추적

**쿼리 단어**: `법인화`, `이후`, `서울대`, `재정`, `변화`

**prefilterScore**:

| 위키 | 매칭 | 점수 | 비고 |
|---|---|---:|---|
| finance | "법인화", "재정", "재원구조분석" 등 | ~15 | preScore +5 |
| board | "법인", "재정", "예산-결산" | ~10 | preScore +5 |
| senate | "법인회계-예산", "재정" | ~6 | |
| plan | "재정", "재정현황" | ~5 | |
| status | — | 0 | **alwaysContext** |
| history | "법인화"가 본문에 있을 수 있음 | ~1 | content 빈도만 |
| vision | "재정전략" | ~3 | |
| yhl-speeches | — | 0 | |

**선택**: finance, board, senate, plan, vision, status (6개) — `MAX_WIKIS = 6` 한도 도달

### Stage B — finance 위키 내부

**fact 페이지 매칭** (분할 안 함, 전체 통째):

| Fact ID | Title | category | 점수 |
|---|---|---|---:|
| `법인회계-운영계산서.fact` | 법인회계 운영계산서 (2017~2024) | 수입·비용 | content 빈도 + category 매칭(+3) |
| `재원구조-분석.fact` | 재원구조 분석 (2017~2024) | 수익구조 | 빈도 (높음) |
| `등록금정책분석.fact` | 등록금 정책 분석 (2017~2026) | 정책 | 빈도 |

**source 청크**:
- 운영차익 분석 source 청크 (법인화 이후 비교 부분)
- ...

### Stage C — LLM 입력

```
### [재무정보공시] 관련 자료

## [fact] 법인회계 운영계산서 (2017~2024) (법인회계-운영계산서.fact) | category: 수입·비용 / years: 2017~2024
{표 + 분석 전체}

---

## [fact] 재원구조 분석 (2017~2024) (재원구조-분석.fact) | category: 수익구조 / years: 2017~2024
{표 + 분석 전체}

---

## 운영차익 구조 분석 (운영차익-분석) | 회의일: 2024-12
{청크 1}

---

### [이사회] 관련 자료
...
```

→ **fact 페이지는 통째로** LLM에 들어가서 답변에 표 그대로 인용 가능. (실제로 사용자가 이전에 받은 답변에서 표가 정확히 나온 이유)

## 📖 예시 3 — "관악 캠퍼스 70년" (메타 빈약 위키 약점)

### Stage A 추적

**쿼리 단어**: `관악`, `캠퍼스`, `70년`

**prefilterScore**:

| 위키 | 매칭 | 점수 | 비고 |
|---|---|---:|---|
| senate | "캠퍼스-인프라", "관악수목원" | ~5 | |
| board | "캠퍼스" | ~3 | |
| vision | "캠퍼스" | ~3 | |
| plan | "캠퍼스" | ~3 | |
| **history** | (없음 — 키워드는 통사편/운영편/교육편/연구편/학생편/공간편 6개뿐) | **0** | ⚠️ |
| status | — | 0 | alwaysContext |
| yhl-speeches | — | 0 | |
| finance | — | 0 | |

### ⚠️ 문제점

쿼리에 "관악", "70년"이 명확히 들어있는데도 **history 위키가 라우팅에서 누락됨**.
- history의 키워드는 `통사편`, `운영편` 같은 챕터명만 있음
- source.tags / source.topics / source.entities 필드가 비어있어서 preScore도 false
- concept-index에도 history 항목 거의 없음 (ingest 단계에서 entity/topic 추출이 0이라서)

**유일하게 잡히는 케이스**:
- `통사편 70년` 처럼 챕터명을 직접 언급하면 prefilter 매칭

→ history 위키는 자연어 질문에서 누락 위험 큼.

### 보완 옵션

| 옵션 | 효과 | 비용 |
|---|---|---|
| **A**. Obsidian source frontmatter에 tags/topics 백필 (`태그: [관악, 1946-1975, 신입생]` 등) | 라우팅·스코어링 모두 정상화 | 데이터 작업 多 |
| **B**. `alwaysContext: true` 추가 | 모든 질문에 history 항상 포함 | 토큰↑ (chunk 5개) |
| **C**. overview를 keyword 추출 대상 강화 | 일부 개선 | 빌드 스크립트 수정 |

## 📖 예시 4 — "전체적인 서울대 현황 알려줘"

### Stage A — Tier 0 발동

`전체적` 이 globalKeyword에 매칭 → **8개 위키 모두 사용** (Tier 0 분기)

```typescript
if (hasGlobalKeyword) {
  const contexts = await Promise.all(
    agents.map(a => a.getContext(query, userRole, true))  // isGlobal=true
  );
  return { ..., isGlobal: true };
}
```

### Stage B — 각 위키 isGlobal 모드

`isGlobal=true`이면 chunkCap이 다르게 적용됨:
```typescript
const chunkCap = options.chunkCap ?? (
  isGlobal ? allowedSources.length        // ← 모든 source 1청크씩
    : guaranteedIds.size > 0 ? MAX_CHUNKS_ENTITY : MAX_CHUNKS
);
```

→ 위키별로 각 source의 첫 청크씩 모두 포함 (광범위 cover)

→ 그러나 `TOTAL_CHUNK_BUDGET = 30`이 우선이므로, router에서 위키별 cap을 다시 분배:
- 8개 위키 모두 사용 → 위키당 평균 ~4청크 (일부 status는 5)

### Stage C — LLM 입력

```
### [평의원회] 관련 자료
{senate 4개 source의 대표 청크}

### [이사회] 관련 자료
{board 4개 source의 대표 청크}

(8개 위키 모두 이런 식으로 ...)
```

답변 톤은 "광범위 요약" 모드 — 각 위키에서 핵심만 1~2줄.

## 📖 예시 5 — Lens 모드: 후보 lens (이석재 후보 시각)

admin이 + 버튼 → "후보 lens 모드 (이석재 후보)" 선택 후 질문:
> "AI 시대 대학의 역할은 어떻게 봐야 할까?"

### 동작 흐름

```
mode: 'lens:leesj' 로 /api/chat 호출
  ↓
1. routeQuery — leesj 위키는 lensPersona이라 자동 제외
   → 일반 라우팅 (yhl-speeches, plan, vision, status... 선택)
  ↓
2. loadPersonaContext('leesj', 쿼리, 'admin')
   → leesj.json에서 stance 9개 중 점수순 매칭
   → "AI시대대학역할.이석재.stance" (topic 매칭 +5)
   → "학부교육혁신.이석재.stance" (content 빈도 부스트)
   → ...
  ↓
3. buildLensSystemPrompt
   → 시스템 프롬프트에 "이석재의 시각으로 분석" + stance 자료 통째 삽입
  ↓
4. LLM 호출
```

### LLM 입력 (system prompt 끝부분)

```
## 🎯 Lens 모드 — 이석재 후보의 시각으로 분석

위 자료들을 이석재 후보의 시각으로 해석·답변하세요.

### Lens 적용 원칙
1. 명시적 입장 우선: 이석재 후보가 직접 표명한 입장 인용. 출처: [이석재 후보-stance] {stanceId}
2. 추론 표시 의무: 명시적 입장 없는 주제는 가치 우선순위에 비추어 추론하되, "(이석재의 명시적 입장은 자료에 없으나, ~의 가치관에 비추어 보면...)" 명시
3. 자료 외 생성 금지 (P1)
4. 일반 자료 인용은 [위키명] 문서ID 형식 그대로
5. 자연스러운 톤 — 헤더 자동 삽입 X

### 이석재 후보의 입장 자료
## [이석재 후보-stance] AI시대대학역할 — 이석재의 입장 (AI시대대학역할.이석재.stance) | topic: AI시대대학역할
{stance 본문 — 핵심 입장, 근거 발언, 맥락}

---

## [이석재 후보-stance] 학부교육혁신 — 이석재의 입장 (학부교육혁신.이석재.stance) | topic: 학부교육혁신
{stance 본문}
```

### 답변 형태

답변에 `[이석재 후보-stance] AI시대대학역할.이석재.stance` 같은 출처 인라인 표기 + `[유홍림총장연설]` 같은 일반 위키 출처가 섞여서 나옴 → **lens 출처와 일반 자료 출처가 한눈에 구분됨**.

| `data.entities[].name` | keywords로 자동 등록 | guaranteedIds + entity 블록 | ✅ (entity.content) |
| `data.entities[].sources` | — | guaranteedIds로 → source 부스트 | ❌ |
| `data.entities[].content` | — | — | ✅ |
| `source.topics` | preScore 매칭 | source 점수 +3 | ❌ (메타데이터만) |

| `source.tags` | preScore 매칭 | source 점수 +2 | ❌ |

| `stance.holder` | preScore 매칭, keywords 등록 | stance 점수 +3 | ❌ (메타데이터만) |

| `stance.content` | — | 단어 빈도 점수 | ✅ (통째) |
| `fact.category` | preScore 매칭, keywords 등록 | fact 점수 +3 | ❌ (메타데이터만) |
| `fact.content` | — | 단어 빈도 점수 | ✅ (통째) |
| `overview.편` | keywords 등록 | overview 점수 +3 | ❌ (메타데이터만) |
| `overview.content` | — | 단어 빈도 점수 | ✅ (통째) |
| `concept-index.json` | forcedWikis + guaranteedPages | guaranteedIds | ❌ (서버 전용) |
| `agents.config.json keywords` | prefilterScore | — | ❌ (서버 전용) |

## ⚙️ 청크 예산 분배 상세

```typescript
const TOTAL_CHUNK_BUDGET = 30;   // 전체 위키 청크 합산 한도
const ALWAYS_CONTEXT_CAP = 5;    // alwaysContext 위키 1개당 5청크 고정
const MAX_WIKIS = 6;
```

**계산식**:
```
refCount = alwaysContext 위키 수 (현재: status 1개 → 5청크)
normalCount = MAX_WIKIS - refCount
normalCap = floor((TOTAL_CHUNK_BUDGET - refCount * ALWAYS_CONTEXT_CAP) / normalCount)
          = floor((30 - 1*5) / 5)
          = floor(25/5)
          = 5청크 / 일반 위키
```

→ 6 위키 활성화 시 → status 5 + 일반 5×5 = **30 청크**

## 🛠️ 개선 가능 포인트

### A. 청크 깊이 (소스당 정보량 ↑)

| 옵션 | 변경 | 위키당 청크 | 토큰 |
|---|---|---|---|
| **현재** | TOTAL_CHUNK_BUDGET 30 | 5 | ~10K |
| **균형** | 30 → 50, 소스당 2청크 | ~9 | ~16K |
| **공격적** | 30 → 70, 소스당 무제한 | ~12 | ~20K |

### B. history 위키 강화

- Obsidian 데이터에 frontmatter 추가 (가장 근본적)
- `alwaysContext: true` 부여 (가장 빠른 임시 해결)
- overview tags 추출 강화 (중간)

### C. Topic 본문도 LLM에 노출

현재 topic.content는 LLM이 못 봄. 만약 topic이 자체 정리 정보를 담고 있다면, 매칭된 topic을 entity 블록처럼 통째로 LLM에 전달하는 옵션 가능. (구현 필요)
