| | |

## 1. 목표

검색된 거버넌스 사실을 토대로 **공약 설계용 인사이트**(찬반·쟁점·전망·구조적 진단·구체 제안)를 생성한다. 핵심 원칙:
- **사실 = 인용에 묶임** (내부 `[N]`, 외부 `[제목](URL)`).
- **의견·제안 = 명시 라벨** (해석/제안/외부지식 구분).
- **외부지식 = 모델 기억이 아니라 *도구검색*** (환각 통제 + 인용 퀄리티).

---

## 2. 아키텍처 — 에이전트형 (단, 기존 스트림 재사용)

### 2.1 구현 방식 — **X 확정 (2026-06-03)**: 사전검색 + 서버 web_search

| | **X. 사전검색 + 서버 web_search (추천)** | Y. 완전 tool-use 루프 |
|---|---|---|
| 내부 사실 | 라우팅으로 *사전 검색* 후 컨텍스트 주입(현 fact 방식) → `[N]` | 모델이 `internal_retrieve` 도구를 *스스로* 호출 |
| 외부 지식 | Anthropic **서버 web_search 도구**(max_uses=2, 자동 URL 인용) — 모델이 필요 시 호출 | 동일 |
| 제어흐름 | **단일 스트림 1회** (web_search는 서버가 mid-stream 처리) — 현 chat route 패턴 그대로 | 다중 라운드 tool-use 루프(우리가 오케스트레이션) |
| 에이전트성 | 모델이 *외부검색 여부·쿼리를 스스로 결정* = 충분히 에이전트적 | 내부검색까지 모델 제어 = 최대 |
| 비용/복잡도 | ~$0.20–0.25, 재사용 최대 | 가변·높음, 신규 루프 구현 |
| start-simple | ✅ | ✗ (v2 승급용) |

> **추천 = X.** web_search가 이미 *서버 도구*(Anthropic 실행, 자동 인용)라 단일 스트림 안에서 "모델이 알아서 외부검색"이 됨 → 에이전트성·외부인용퀄리티 둘 다 충족하면서 현 인프라 재사용. 내부검색까지 모델이 제어하는 Y의 이득은 작음(우리 라우팅이 이미 양호). **Y는 X가 부족할 때 v2.**

### 2.2 구조 (Option X)

```
공약설계 모드 (mode='policy', admin, Sonnet)
 ① 사전 검색: routeQuery + enforceContextBudget → 내부 사실 [N] 주입 (fact 파이프라인 재사용)
 ② 단일 스트림 생성:
     system = buildPolicySystemPrompt (하이브리드 grounding + 라벨 + 출력구조)
     tools  = [ WEB_SEARCH_TOOL(max_uses=2) ]   ← 서버도구, 모델이 외부 필요시 호출→URL 자동인용
 ③ 메모리: summary-buffer (최근 턴 원문 + 오래된 턴 요약 + 예산상한)
 ④ 출력: 사실=[N] / 외부=[제목](URL) / 해석·제안=라벨
```

## 4. 인용 모델 — 두 네임스페이스

- **내부 사실**: 기존 번호인용 `[N]` (buildNumberedContexts → resolveText 재사용). 출처 = 위키 source.
- **외부 지식**: web_search 결과 → `[제목](URL)`. Anthropic 서버도구가 인용 메타 반환 → 본문에 마크다운 링크.
- **분리 렌더**: 사용자가 "우리 자료(N)"와 "외부(URL)"를 즉시 구분. 섞지 않음.

---

## 5. 시스템 프롬프트 (`buildPolicySystemPrompt`)

fact 프롬프트(P0~P7) 위에 **공약설계 레이어** 추가:
1. **역할**: 거버넌스 사실을 토대로 공약 설계 인사이트(찬반·전망·제안) 생성.
2. **grounding 하이브리드(D1)**: 사실 주장은 `[N]`/URL 필수. 사실 위의 *해석·평가·전망·제안*은 허용하되 **명시 라벨**.
3. **외부지식 규칙**: 위키에 없는 사례·벤치마크는 **web_search로 검색해 URL 인용** — *기억으로 단정 금지*(환각/스핀 차단, D4).
4. **라벨 규약**: `(해석)` `(제안)` `(외부지식)` 또는 섹션 구분.
5. **출력 구조**(아래 §6) 유도.
6. **P1 유지**: 검색·도구로 확인 안 된 사실은 만들지 않음.

## 6. 출력 구조 (목표 산출물)

```
📊 사실 근거       내부 [N] / 외부 [제목](URL)
⚖️ 찬반·쟁점       (해석) 양측 논거
🔮 전망            (해석) 근거 기반 시나리오
💡 제안            (제안) ①…[근거] ②…(외부지식 [URL])
```
질문 성격에 따라 섹션 가감(완전 고정 X). 사실 없는 섹션은 생략.

---

## 7. 메모리 — summary-buffer (D8)

- **최근 K턴 원문** 유지 + **그 이전 턴은 재귀 요약**("지금까지 논점: …")으로 압축.
- **토큰 예산상한**(FR7): 토론 컨텍스트가 상한 초과 시 가장 오래된 것부터 요약·축출.
- 구조화 "토론상태 스크래치패드"(입장·합의·미해결 객체)는 **v2 옵션**.
- 교차세션 사용자기억은 범위 밖.

## 8. 비용 모델

| 항목 | 비용 |
|------|------|
| 내부 검색(사전) | Voyage ~$0.001 (8×임베딩 중복은 별도 최적화 대상) |
| 메인 생성 | 입력~10–15k + 출력 → ~$0.10 |
| web_search ≤2회 | ~$0.08×2 = ~$0.16 (발동 시) |
| **건당 상한** | **~$0.25** (외부검색 2회 다 쓸 때). admin·저빈도라 허용. |

> fact 모드와 달리 *여기가 돈 쓸 자리* — 단 web_search **max_uses=2로 하드 바운드**.

## 10. 환각·스핀 가드 (D4 일반화)

- 사실 주장 → 반드시 `[N]`/URL. 출처 없는 사실 서술 금지.
- 외부지식 → web_search 경유만 (파라미터 기억 단정 금지).
- 해석·제안 → 라벨로 사실과 분리.
- 인용 검증(citation-validator 기계) 재사용 — 내부 `[N]`이 실제 source에 매핑되는지.

---

## 11. UI

- 모드 배지(`💡 공약설계`) + (Phase3) 오버라이드.
- 사실/해석/제안/외부 시각 구분(라벨·색). 외부는 URL 링크칩.
- web_search 발동 표시(비용 인지).

---

## 12. 검증 (D6 — 사용자 담당)

- **실제 캠프형 질문**으로 사용자가 루브릭 인간평가(근거성·실현가능성·창의성·사실정확성). 합성 질문 금지.
- 회귀: fact 모드 영향 0 (policy는 별도 mode라 기존 경로 불변).

## 14. Open / Risks

- **O1**: ✅ 해결 — X 확정(사전검색 + 서버 web_search).
- **O2**: summary-buffer의 K(원문 유지 턴 수)·요약 트리거 임계 — Do에서 측정 튜닝.
- **R1**: web_search 결과 품질 편차 → 프롬프트로 "신뢰 출처 우선" 지시 + 사용자 검증.
