> 작성일: 2026-06-03 · 상태: **plan (요구사항 합의 중)** · feature: `multi-agent-workflow`
> 상위 우산 플랜. 하위 컴포넌트(2.2 공약설계 / 2.3 프레이밍 / 라우터)는 각 Design 단계에서 별도 문서화.

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 현재는 fact 답변 단일 모드뿐. 총장 캠프의 **공약 개발자**(인사이트·제안 필요)·**캠페인 담당**(청중별 메시지 프레이밍 필요)는 "사실 나열"만으론 일을 못 함. 모드를 수동 토글하는 것도 번거로움. |
| **Solution** | 하나의 **지속 성장 지식 패브릭**(정적 위키 + 웹검색 + DB/live) 위에, 출력 성격이 다른 **3개 에이전트**(2.1 fact / 2.2 공약설계 / 2.3 프레이밍)를 얹고, **상위 라우터**가 질문 의도를 판정해 적합 에이전트(혹은 체인)로 자동 전달. |
| **Function·UX 효과** | 사용자는 토글 없이 질문만 → 라우터가 "이건 사실조회/공약설계/프레이밍" 판정 → 알맞은 깊이의 답. 공약 개발자는 근거 기반 **제안·찬반·전망**을, 캠페인 담당은 **단과대/분야별 프레이밍**을 얻음. |
| **Core Value** | 같은 검증된 사실 위에서 *역할별 산출물*을 생성 → 환각을 사실에 묶은 채 fact→인사이트→메시지로 **가치사슬 확장**. 비싼 생성은 *해당 의도일 때만* 발동돼 비용 통제. |

## 1. 배경 & 목적

- **이미 구축**: 2.1 fact 답변 에이전트(거버넌스 9위키 + 단과대), 공유 검색층(routeQuery·복잡도예산·RRF·rerank·번호인용), lens 모드(인물 시각, admin), 웹검색 폴백, 단과대 Tier3/4(structured_facts/live_cache).
- **지식층은 정적이 아님**: 데이터는 **계속 추가**되고, 실시간성·비-SNU 정보는 **웹검색 또는 DB**(단과대 위키 Tier4 패턴)에서 동적으로 끌어옴. → 지식층 = "성장하는 다중소스 검색 패브릭".
- **목표**: fact 단일 모드를 **역할 특화 다중 에이전트 + 의도 라우터**로 확장. 단, fact 토대를 훼손하지 않고(②.²/②.³은 fact 위 *출력 레이어*), 비용·환각을 통제하며, 단계적으로.

## 2. 요구사항

### 2.1 기능 요구사항(FR)
- **FR1** 지식층: 정적 위키(벡터/키워드) + 웹검색 폴백 + DB/live_cache(Tier4)를 **단일 검색 인터페이스**로 통합, 출처 종류를 산출물에 표기.
- **FR2** Agent 2.2 공약설계: 검색된 사실을 토대로 **찬반·전망·제안·구조적 진단**을 생성. 사실=인용[N], 의견·제안=명시 라벨(추론 표시 의무, lens 기계 재사용).
- **FR3** Agent 2.3 메시지 프레이밍: 사실/공약을 **임의 청중별 프레이밍**으로 변환. 청중은 단과대·분야에 **한정되지 않음** — 학생/교원/직원/동문/언론/정부·기업/유권자 그룹/특정 이해관계자 등 **지정 가능한 모든 대상**. 사실 왜곡 금지(프레이밍 ≠ 날조), "프레이밍" 라벨 명시.
- **FR4** 상위 라우터: 질문 의도(fact / 공약설계 / 프레이밍)를 판정해 **자동 분기**. 필요 시 **체인**(fact→공약→프레이밍 일부만 태움). 선택된 에이전트를 사용자에게 노출 + **수동 오버라이드** 허용.
- **FR5** 비용 가드: 에이전트별 컨텍스트 예산·max_tokens 상한. 라우터가 의도를 잘못 키워 비싼 에이전트를 남발하지 않도록 보수적 분기 + 폴백.
- **FR6** 메모리(공약설계/토론): **5턴 단기기억으로 부족**. 토론은 누적 논점·입장·반론이 핵심이라 **확장 메모리**(전체 대화 보존 / 직전 턴 요약 장기기억 / "토론 상태" 스크래치패드) 필요. 에이전트별 메모리 정책 분리(fact=5턴 유지, 공약설계=확장).
- **FR7** 토론 토큰 제어: 토론 모드는 턴 누적으로 컨텍스트 팽창 → **이전 턴 요약·압축 + 예산 상한**으로 비용·일관성 균형. 무한 누적 금지.

### 2.2 비기능 요구사항(NFR)
- **NFR1 환각 통제**: 모든 의견·제안·프레이밍은 인용된 사실에 grounding. 사실/해석/제안을 시각적으로 구분.
- **NFR2 비용**: 쿼리당 의도별 상한(예: fact ≤$0.15 / 공약설계 ≤$0.30 / 프레이밍 ≤$0.20) — Design에서 측정·확정.
- **NFR3 모듈성**: 각 에이전트 = 시스템프롬프트+생성행동 차이. 검색층·인용층 공유. 신규 에이전트 추가가 O(1)에 가깝게.
- **NFR4 평가**: 실제 질의로그(합성 X) 기반 회귀·품질 비교 유지.

## 3. 아키텍처 개요

```
[지식 패브릭 (지속 성장 · 다중소스)]
  ├─ 정적: 위키 JSON (벡터 pgvector + 키워드) — governance 9 + 단과대
  ├─ 라이브: 웹검색 폴백 (Claude web_search) — 외부·최신·비교
  └─ 구조: DB / live_cache (Tier3/4) — 연락처·통계·최신공지
                    │
        [공유 검색층: routeQuery + 복잡도예산 + RRF/rerank + 번호인용]  ← 이미 구축
                    │
   ┌──── [상위 라우터: 의도 판정 → 분기 / 체인 + 오버라이드] ────┐   ← Phase 3
   │                      │                         │
 2.1 fact            2.2 공약설계               2.3 프레이밍       ← Phase 1·2
 (구축됨,            (찬반·전망·제안,            (단과대/분야별
  P7로 강화)          추론 라벨)                  메시지, 라벨)
                    └──── 출력 레이어: 같은 fact 검색 위에 얹힘 (사일로 아님) ────┘
```

**핵심 원칙**: 검색은 공유, **에이전트는 출력 변환**. 2.2·2.3은 2.1이 끌어온 *동일한 인용 사실*에 제안/프레이밍을 더함 → grounding 유지 = 환각 1차 방어.

## 4. 컴포넌트별 설계 방향 (개략 — 상세는 각 Design)

### 4.0 에이전트 구성요소 — multi-agent 완결성 점검 (★ 누락 점검 필수)

각 에이전트는 "프롬프트"가 아니라 **{모델 · 툴 · 메모리 · 오케스트레이션}** 4요소를 갖춘 단위. Design 시 각 칸을 명시·검증하고 **빈칸 = 미완 에이전트**로 본다.

| 요소 | 2.1 fact | 2.2 공약설계 | 2.3 프레이밍 | 상위 라우터 |
|------|---------|-------------|-------------|------------|
| **모델** | Sonnet | 추론 깊이 최우선 (Sonnet/Opus 검토) | 변환 중심 → 비용↓ 모델 가능(Haiku?) | 코드분류 우선, LLM분류는 최후(저비용) |
| **툴** | 검색·인용·web_search·Tier4 | +타대학 벤치마킹 web_search, 수치 계산 | 프레이밍 변환(외부툴 최소) | 분기/체인 디스패치, 폴백 |
| **메모리** | 5턴 단기 | **확장**: 토론상태·요약 장기기억(FR6) | 대상 청중 컨텍스트 + 직전 산출물 | 라우팅 이력·오버라이드 |
| **오케스트레이션** | 단독 | 라우터 하위 + 체인 입력(fact 결과) | 라우터 하위 + 체인 입력(fact/공약) | **최상위** — 의도판정·체인구성·폴백·오버라이드 |

> 멀티에이전트 구조이므로, 한 에이전트라도 4요소 중 빠지면(예: 토론인데 메모리 5턴 그대로) 역할 미달. P1~P3 Design에서 이 표를 체크리스트로 사용.

### 4.1~ 컴포넌트 요약

| # | 컴포넌트 | 핵심 | 재사용 | 신규 |
|---|---------|------|--------|------|
| 4.1 | 지식 패브릭 다중소스화 | static+web+DB 단일 인터페이스 | searchVector·web_search·Tier4 | 소스 통합·출처라벨 |
| 4.2 | Agent 2.1 fact | 현행 + P7/단과대격리 | 전부 | (강화만) |
| 4.3 | Agent 2.2 공약설계 | 사실→제안·찬반·전망 | 검색·인용·lens 추론라벨 | `buildPolicySystemPrompt`, mode |
| 4.4 | Agent 2.3 프레이밍 | 사실/공약→**임의 청중**별 메시지 | 검색·인용 | `buildFramingSystemPrompt`, **자유지정 청중 파라미터** |
| 4.5 | 상위 라우터 | 의도분류→분기/체인 | classifyComplexity 패턴 | 의도분류기, 체인 오케스트레이터, 오버라이드 UI |

### 4.6 UI 표현 (개략 — Design에서 구체화)

- **라우터 투명성**: 선택된 에이전트 배지 + 1클릭 오버라이드(fact ↔ 공약 ↔ 프레이밍).
- **사실/해석/제안 시각 구분**: 라벨·색·블록 분리 (예: 사실=인용칩 `[N]`, 제안=점선블록, 외부지식=별도 태그, 프레이밍=청중태그).
- **토론 모드 UI**: 멀티턴 누적 → 찬반 컬럼 / 입장 타임라인 / 토큰 사용량 표시(FR7 가시화).
- **프레이밍 UI**: 자유 청중 선택·입력 + (선택) 복수 청중 프레이밍 나란히 비교.
- **비용 인지**: 비싼 모드(공약설계) 진입 시 사용자 신호(선택적).

| **D2** | **체인** | fact→공약→프레이밍 중 필요한 단계만 순차 |
| **D3** | **신뢰도 임계 + fact 폴백** | ⚠️ **라우터 정확성이 관건** → 의도분류 정확도를 SC로 측정·튜닝, 임계 미달 시 안전하게 fact. (보완: 선택 에이전트 배지 노출 권장) |
| **D4** | **가드 필수** | 사실앵커 + "프레이밍" 라벨 + 날조 금지 |
| **D5** | **admin 전용(1차) → tier1 확대(추후)** | 권한 게이트 단계적 오픈 |
| **D6** | **사용자 직접 구축 + 인간평가 사용자 담당** | 자동 평가 미구축 (루브릭 인간평가는 사용자가) |
| **D7** | **전부 Sonnet (1차)** | 품질 아쉬우면 공약설계 모델 업그레이드 검토 |
| **D8** | **요약 장기기억 (summary-buffer 하이브리드)** | §6.2 리서치 근거. 구조화 토론상태는 v2 옵션 |

### 6.2 D8 리서치 — 일반 LLM의 대화 메모리 방식 (2026-06)

업계 표준은 **다층 메모리**: ① 단기 버퍼(최근 턴 원문) → ② 요약(오래된 턴 압축) → ③ 장기 아카이브(retrieval). 핵심 시사점:
- **ChatGPT는 RAG를 안 씀** — 최근 대화 *요약 리스트* + 압축된 사용자기억을 매 메시지에 주입. "복잡한 retrieval보다 단순 요약이 나을 때가 많다"가 교훈.
- **summary-buffer 하이브리드**(최근 N턴 원문 + 그 이전 요약)가 가장 실용적 표준(LangChain `ConversationSummaryBufferMemory` 등).
- **벡터 retrieval 메모리**(대화이력 RAG)는 *초장기·교차세션*엔 강하나 단일 세션 토론엔 과함(지연·해석불가·복잡도).
- **베스트프랙티스 = "단순하게 시작, 필요할 때 확장"**.

→ **결정**: 단일-세션 토론엔 **summary-buffer 하이브리드**(최근 턴 원문 + 오래된 턴 재귀요약 + 예산상한 = FR7)가 정답. 토론 입장/논점을 구조화한 "토론상태 스크래치패드"는 효과 크지만 **v2 옵션**(start-simple 원칙). 교차세션 사용자기억(ChatGPT식)은 현재 범위 밖.

## 7. Success Criteria

- **SC1** 2.2가 실제 공약형 질문에서 *사실 인용에 묶인* 제안·찬반·전망을 생성, 환각 0(인용 검증).
- **SC2** 2.3이 청중별 프레이밍을 생성하되 사실 왜곡 0, "프레이밍" 라벨 100%.
- **SC3** 라우터가 실질문 셋에서 의도 분기 정확도 목표치 달성(Design에서 수치 확정) + 오버라이드 동작.
- **SC4** 쿼리당 비용이 의도별 상한 내(측정).
- **SC5** fact 모드 회귀 없음(기존 품질 유지).

## 9. 비용 모델 (개략 — Design에서 측정 확정)

- 공유 검색층은 동일 → 추가비용은 **생성 토큰 차이**.
- 2.2 공약설계: 더 긴 컨텍스트+출력 추론 → fact 대비 ~1.5–2×. 상한 ~$0.30 가정.
- 2.3 프레이밍: 사실/공약 입력 + 짧은 변환 출력 → fact와 유사~소폭. 상한 ~$0.20.
- 라우터: 의도분류는 **코드 기반(저비용) 우선**, LLM 분류는 최후. 분류 자체 비용 ~$0.
- 핵심: 비싼 에이전트는 *의도 맞을 때만* 발동 → 평균비용은 질문 분포에 의존. Design에서 실분포로 검증.
