## 1. 🔴 HIGH — 즉시 수정 (보안 / 데이터 정합성, 전부 검증 확정)

### H-1. 모든 admin이 무제한 권한 승격 + 마스터 삭제 가능
- **위치**: `app/api/admin/users/route.ts:22-65`
- **문제**: PATCH가 `role:'admin'`을 허용 → 어떤 admin이든 타인을 admin으로 승격, 다른 admin/마스터/자기 자신을 hard-delete 가능. 마스터 보호·self-action 가드 전무.
- **수정**: 역할관리와 승인 분리. admin 부여는 마스터 전용(`session.user.id==='master-admin'`)으로 게이트, `userId==='master-admin'` 대상 변경/삭제 거부, 다른 admin 강등·자기 거부 차단.

### H-2. 세션 role이 DB와 절대 동기화되지 않음 (최대 8시간 staleness)
- **위치**: `lib/auth/config.ts:96-108` (+ `session.maxAge=8h` line 12)
- **문제**: `jwt` 콜백이 로그인 시점에만 `token.role`을 굳히고(`if (user) token.role = user.role`) 이후 모든 인가(미들웨어·API)가 토큰만 신뢰. **거부(삭제)된 사용자도 JWT 만료까지 접근 가능**, 강등/승인이 재로그인 전까지 미반영. (커밋 `aeaadbe`에서 504 회피로 세션 DB조회를 제거한 부작용.)
- **수정**: jwt 콜백의 `!user`(후속 호출) 경로에서 TTL 기반으로 DB의 role 재조회, 행이 없으면 강제 로그아웃 신호. 또는 maxAge 대폭 단축.
- **✅ 적용됨 (2026-05-29)**: jwt 콜백에 **5분 TTL 스로틀 DB 재동기화** 추가(`lib/auth/config.ts`). 매 요청 DB조회로 인한 504 회귀(`aeaadbe`) 회피 + 강등/거부 사용자를 ~5분 내 반영(삭제 사용자는 `pending`으로 강등). DB 오류 시 기존 토큰 유지(가용성 우선).

### H-3. `GET /api/conversations/[id]`에 소유권 검사 없음 — 아무 인증 사용자가 남의 대화 전문 열람

- **문제**: "공개 뷰어 지원" 의도로 ownership 검사를 생략하고 content/sources/routedAgents/mode 전체 반환. 정작 같은 파일의 DELETE(51-59)와 chat POST(`route.ts:60-75`, "보안 floor")는 소유권을 엄격히 검사 → 모순적 보안 구멍. tier2도 열람 가능.

**구체적 예시 (왜 단순 패치가 안 되는가)**

- **시나리오 ② 개인 질문 유출**: conversation id는 UUID라 추측은 불가하지만, `/public`이 그 id 목록 + title(질문 앞 50자)을 제공 → 누구나 타인의 질문 전문 + 라우팅 위키 + lens 모드까지 열람 가능.

**수정 옵션 (택1, 결정 필요)**

| 옵션 | 내용 | 기능 영향 |

### H-4. 스트림 중간 실패 시 화면의 부분 답변이 통째로 에러문구로 교체 + DB에 고아 user 메시지

- **문제**: user 메시지는 스트림 시작 전 저장(127-135), assistant는 스트림 성공 시에만 저장(258-266). for-await 루프나 retry 중 throw 시 → error 이벤트만 송신 → 클라가 누적 청크를 에러문구로 덮어씀 + assistant 없는 user행이 DB에 잔류. 다음 턴에 user-user 연속이 되어 **Anthropic API가 후속 요청을 거부**할 수 있음.

- **✅ 적용됨 (2026-05-29)**: `route.ts` — 가드된 `send`(연결 끊김 시 `closed`) + 루프 조기 종료(`req.signal.aborted`) + 스트림 실패 시 부분답변을 경고 마커와 함께 DB 저장(고아 user행 차단) + `keepContent` 비파괴 에러. `ChatPage.tsx` — `keepContent`면 스트리밍 내용 보존하고 경고만 덧붙임. M-13(주석 불일치)도 함께 정정.

### H-5. 괄호 포함 제목이 `[N]` 인용 주입을 깨고 실제 source ID를 LLM에 노출
- **위치**: `lib/llm/citations.ts:45-53, 75-91`
- **문제**: 헤더 `## 제목 (id) | ...`에서 정규식이 **첫 번째** 괄호를 id로 캡처(`([^(\n]*?)\(([^)]+)\)`). 그런데 실제 source 제목의 ~8.7%(173개 중 15개)가 괄호를 포함 → id 매핑 실패 + sid가 LLM에 그대로 노출(번호화 우회) + wrong-attribution.
- **수정**: id 캡처를 라인의 **마지막** 괄호 그룹으로 앵커링(title 그룹을 greedy로).
- **✅ 적용됨 (2026-05-29)**: `citations.ts`의 두 패턴(titleExtract/header)을 비탐욕 title + ` | `/줄끝 경계 단언으로 변경 → 제목·메타에 괄호가 있어도 진짜 id만 캡처. 전용 검증 4/4 통과, `resolveText` 회귀 테스트 무변화.

### H-6. 글로벌 키워드가 부분문자열 매칭 → 전체 위키 full-coverage 오발동
- **위치**: `lib/agents/router.ts:115-121`
- **실제 키워드**(`data/agents.config.json:1008-1015`): `["전체", "전체적", "전반적", "모든", "종합", "총장이 알아야 할"]`
- **문제**: `globalKeywords.some(kw => queryLower.includes(kw))` — 부분문자열 매칭이라 키워드가 무관한 단어 *안에* 들어가도 9개 위키 전부 무캡 호출(`getContext(query, role, true)`). 오답 + 9회 벡터검색·풀콘텐츠 비용.

**구체적 예시 (부분문자열 오발동)**
| 입력 쿼리 | 매칭 | 결과 |
|----------|------|------|
| "**안전체계**는 어떻게 운영되나요?" | `'전체'` ⊂ `안전체계` | ❌ 9개 위키 전부 호출 (전혀 global 아님) |
| "**종합운동장** 신축 계획?" / "**종합감사** 결과?" | `'종합'` ⊂ `종합운동장`/`종합감사` | ❌ 전체 호출 |
| "재정 운영의 **전반적** 추세" | `'전반적'` | △ 의도가 모호 (특정 위키일 수도) |
| "**전체** 위키 종합해서 알려줘" | `'전체'`,`'종합'` | ✅ 올바른 global (보존돼야 함) |

**왜 단순 토큰 일치로는 부족한가 (한국어 형태소)**: `tokens.includes('전체적')`은 조사 결합형 "**전체적으로**"(= 전체적+으로)를 놓침. 반대로 `includes`는 "안전체계"를 잘못 잡음.

**매처별 실측 (14개 라벨 쿼리, 2026-05-29)** — false positive가 false negative보다 비싼 문제(오발동=9위키 호출+노이즈, 누락=일반 라우팅이 대체)라 **정밀도 우선**:

| 매처 | 정밀도 | 재현율 | 비고 |
|------|:-----:|:-----:|------|
| `includes` (현재) | 60% | 100% | 안전체계·종합운동장·종합감사·전체회의 모두 오발동 |
| `startsWith` (초안) | 67% | 100% | infix("안전체계")만 해결, **prefix 복합어("종합운동장"등) 여전히 오발동** → 미흡 |
| `exact token` | 100% | 50% | 오발동 0이나 활용형("전체적으로","전반적인","종합적으로") 누락 |
| **`exact token` + 확장리스트** | **100%** | **100%** | ✅ 권장 |

**결론**: `startsWith`는 효과가 약함(정밀도 67%). 한국어 복합어가 키워드를 **접두**로 자주 가져서(종합운동장·전체회의) 접두 매칭으론 못 거른다. **권장안 = 토큰 정확일치 + 활용형을 포함한 확장 키워드 리스트**:
```ts
// router.ts:115 — 단일 단어는 토큰 정확일치, 다단어 구문은 includes
const tokens = queryLower.split(/[\s,]+/).filter(Boolean);
const hasGlobalKeyword = globalKeywords.some(kw =>
  kw.includes(' ') ? queryLower.includes(kw) : tokens.includes(kw));
```
```jsonc
// data/agents.config.json — globalKeywords에 활용형 추가
["전체", "전체적", "전체적으로", "전반적", "전반적으로", "전반적인",
 "모든", "종합", "종합적", "종합적으로", "총장이 알아야 할"]
```
- **트레이드오프**: 리스트에 없는 새 활용형은 누락(false negative) → 단, 그 경우에도 일반 스코어링이 관련 위키로 라우팅하므로 답변 자체는 생성됨. 리스트는 운영하며 보강.

> 위 `exact token + 확장리스트` 안으로 적용할지(권장), 키워드 세트를 더 손볼지 알려주시면 반영하겠습니다.

### ⚠️ 추가 발견 — pgvector ivfflat 인덱스 degenerate → semanticRoutingHints recall ~0% (2026-05-29 실측, **HNSW로 해결 완료**)
- **위치**: `drizzle/0002_pgvector.sql:26-29`, `0003`(ivfflat lists=100), `lib/embed/search.ts:121-136`(semanticRoutingHints)
- **문제**: ivfflat 인덱스가 *빈 테이블*에 생성돼 centroid 미학습. `ivfflat.probes=1`(기본) + lists=100 → cross-wiki `ORDER BY embedding <=> q LIMIT k` 쿼리가 ~1%만 탐색. **실측 recall@10 = 0%** (probes=1 vs exact). 즉 임베딩 기반 위키 라우팅이 사실상 무작위 결과를 반환 → 키워드/concept-index가 라우팅을 대신 떠받쳐 옴. (per-wiki searchVector는 wiki_id 필터로 seq scan=exact라 영향 없었음.)
- **✅ 해결 (2026-05-29, migration `drizzle/0004_hnsw_index.sql`)**: ivfflat → **HNSW**(m=16, ef_construction=64) 전환. 적용 후 cross-wiki recall@10 **0% → 100%**. 현재 규모(1,086행)에선 planner가 seq scan(exact) 선택, 데이터 증가 시 HNSW 자동 사용.

### ✅ B-2 — 라우팅 정밀도/답변품질 (2026-05-30, 측정 기반)
- **방향 결정(시뮬)**: "선택 위키 줄이기(tightening)"는 **recall 86%→77% 회귀에 집중도 개선 0%**(균등 min-5 예산 floor가 원인) → **폐기**. 대신 **신뢰도 가중 chunk 예산 분배** 채택 — 위키는 그대로 선택(recall 보존)하되, 임베딩 근접도(softmax)+키워드로 예산을 가중 배정.
- **변경**: `lib/embed/search.ts` semanticRoutingHints — 위키별 MIN 직접 집계(**M-5 해결**: top-K LIMIT 전 GROUP BY 굶김 제거) + 전체 위키 거리 반환. `lib/agents/router.ts` — 균등 분배 → 가중 분배(TEMP=0.07, KW_COEF=0.4) + concept/키워드 관련 위키 floor 상향(다중위키 2번째 위키 대비).
- **결과(22 라벨 질문)**: recall **86% 유지**, 관련 위키 집중도 **24%→31%**, 관련 위키당 청크 **5.6→8.6(+54%)**, 총 컨텍스트 33.4(통제). **위키 수 증가에도 확장 가능**(가중이 근접 위키에 집중).
