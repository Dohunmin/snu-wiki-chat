# 스코어링 및 LLM 답변 생성 동작 보고서

> **작성일**: 2026-05-04
> **대상**: SNU 거버넌스 위키 챗 시스템
> **목적**: 사용자 질문이 LLM 답변으로 이어지는 전 과정의 점수 계산 및 선택 로직을 케이스별로 문서화

## 1. 전체 파이프라인 개요

```
[사용자 질문]
   ↓
[1. Router (lib/agents/router.ts)]                ← 어느 위키를 호출할지 결정
   ├── Tier 0: 글로벌 키워드 → 모든 에이전트
   ├── Tier 1: keywords 배열 매칭
   ├── Tier 2: 메타데이터(tags/topics/entities) 경량 스캔
   └── Last-resort: 모두 호출
   ↓
[2. WikiAgent.getContext (lib/agents/wiki-agent.ts)] ← 각 에이전트별로 관련 청크 추출
   ├── (a) Sensitive 필터링
   ├── (b) Entity/Topic 역참조 → guaranteedIds
   ├── (c) 소스 단위 점수
   ├── (d) 청크 단위 점수
   ├── (e) 소스별 대표 청크 우선 선택
   └── (f) chunkCap만큼 자르기 (15 / 30 / 무제한)
   ↓
[3. Confidence 필터 (router.ts)]                  ← context 신뢰도 < threshold면 제외
   ↓
[4. Prompt 빌드 (lib/llm/prompts.ts)]              ← system + user 메시지 구성
   ↓
[5. LLM (Claude Sonnet 4.6) 스트리밍 응답]
```

## 2. 스코어링 단계별 상세

### 2.1 Router 단계

| Tier | 조건 | 동작 |
|---|---|---|
| **Tier 0** | 쿼리에 글로벌 키워드 포함 + Tier 1 미매칭 | 4개 에이전트 전부 호출, `isGlobal=true` |
| **Tier 1** | 에이전트 keywords 배열 중 하나가 쿼리에 포함 | 매칭된 에이전트만 호출 |
| **Tier 2** | Tier 1 미매칭 시 fallback | 각 에이전트의 `preScore` (메타데이터 스캔) 통과한 에이전트만 |
| **Last-resort** | Tier 1·2 모두 미매칭 | 모두 호출 (정보 손실 방지) |

**글로벌 키워드** (data/agents.config.json:routing):
`전체`, `전체적`, `전반적`, `모든`, `종합`, `총장이 알아야 할`

**중요**: Tier 1이 매칭되면 글로벌 키워드는 무시된다. 즉 `이사회 전체 요약`은 board만 호출.

**Confidence threshold**:
- Tier 1 경유: `> 0.3`
- Tier 2 경유: `> 0.4` (낮은 정확도 매칭이므로 더 엄격)
- 필터 후 모두 제거되면 원본 유지 (정보 손실 방지)

### 2.2 WikiAgent 단계

#### (a) Sensitive 필터링

```typescript
allowedSources = sources.filter(s => isSensitiveAllowed || !s.sensitive)
```

| 역할 | sensitive 접근 |
|---|---|
| admin | ✅ |
| tier1 | ✅ |
| tier2 | ❌ |
| pending | ❌ |

**Sensitive 판정 기준** (build-wiki-data.ts:226):
- senate: 본문 또는 태그에 `총장추천위`, `인사-비공개` 포함
- board: `총장-선출`, `이사-선임`, `감사-선임` 포함
- plan, vision: 없음

#### (b) Entity/Topic 역참조 (guaranteedIds)

쿼리 단어가 entity 또는 topic의 이름·id·alias와 부분 일치하면, 해당 entity·topic이 참조하는 **모든 소스**를 `guaranteedIds`에 추가.

```typescript
names.some(n => queryWords.some(w => n.includes(w) || w.includes(n)))
```

이 기준은 **양방향 부분 매칭**이라 매우 관대함. 예:
- `평의원회` 쿼리 → entity `평의원회`, `미래연구위원회`, `상임위원회`, `운영위원회`, `재경위원회` 모두 매칭
- 결과적으로 51개 senate 소스 중 18개가 guaranteed로 끌려옴

#### (c) 소스 단위 사전 점수

쿼리 키워드가 소스의 metadata와 매칭되는지 확인:

| 매칭 대상 | 점수 |
|---|---|
| guaranteedIds에 포함 | +5 |
| topic 매칭 | +3 |
| entity 매칭 | +2 |
| tag 매칭 | +2 |
| content 단어 매칭 | +1 |

`score > 0`인 소스만 `sourcesToProcess`로 선별. 후보가 0개면 모든 allowedSources를 처리.

#### (d) 청크 단위 점수

각 소스를 `## 헤더` 단위로 분할 후 청크별 점수:

```typescript
score = Σ (queryWord 등장 횟수 in chunk)
if (isGuaranteed) score = score * 2 + 1   // entity/topic 매칭 보장 청크는 부스트
```

**핵심 부작용**: guaranteed 소스의 모든 청크에 +1 보너스가 붙어, 텍스트 매칭이 0인 청크도 살아남음. 이는 의도된 동작이지만 chunkCap 안에서 점수 경쟁이 치열해짐.

#### (e) 소스별 대표 청크 선택

```typescript
sorted = scoredChunks.sort((a, b) => b.score - a.score)
firstChunks = []  // 소스별 최초(=최고점) 청크 1개
restChunks  = []  // 나머지

for (c of sorted) {
  if (!coveredSources.has(c.id)) firstChunks.push(c)
  else restChunks.push(c)
}
chunksToUse = [...firstChunks, ...restChunks].slice(0, chunkCap)
```

**의도**: 적은 소스에 청크가 몰리지 않도록 소스별 대표 1개를 먼저 보장.

**부작용**:
- "대표 청크"는 점수 최고 청크이지, **첫 번째 청크(헤더)가 아님**
- 헤더에만 있는 메타데이터(회의일, 참석자)는 종종 손실됨
- ✅ **이미 패치 적용**: source block 헤더에 `| 회의일: YYYY-MM-DD` 추가 (wiki-agent.ts:210)

#### (f) chunkCap 결정

| 조건 | chunkCap |
|---|---|
| `isGlobal=true` | `allowedSources.length` (사실상 무제한) |
| `guaranteedIds.size > 0` (entity/topic 매칭) | 30 |
| 그 외 (일반) | 15 |

## 3. 케이스별 시뮬레이션

다음은 실제 데이터(senate 51건, board 26건, plan 16건, vision 9건)에서 시뮬레이션한 결과.

### Case A: 인물 entity 직격 — `"유홍림 총장 관련 안건"`

```
Router Tier 1: 이사회(1건: 총장), 중장기발전계획(1건: 총장)
  → senate, plan 미매칭 (의도된 동작 — 총장은 이사회 안건)

[board agent]
  matchedEntities: 총장추천위원회, 유홍림
  matchedTopics: 총장 선출
  guaranteedIds: 26개  ← board 전체 26개 모두 끌려옴 (총장 선출 topic이 모든 회의에 등장)
  chunkCap: 30 (entity 매칭이라 boost)
  totalScored: 110개 청크
  finalSources: 26개 ← board 전체 커버
  confidence: 0.8
```

**해석**: 인물 entity 매칭이 강하게 작동하여 board 전체를 활용하지만, **chunk 30개 안에서 26개 소스를 커버하므로 소스당 평균 1.15개 청크**만 LLM에게 전달됨. 소스가 많아지면 깊이가 얕아진다.

---

### Case B: 토픽 매칭 — `"AI 가이드라인 의결"`

```
Router Tier 1: 평의원회(4건), 이사회(1건), 대학운영계획(1건)

[senate agent]
  matchedEntities: (없음)
  matchedTopics: AI 가이드라인, AI 대학원
  guaranteedIds: 4개
  chunkCap: 30
  finalSources (14개): 19기-5차, 18기-21차, 19기-3차, 19기-2차, 19기-4차, ...
  confidence: 0.8
```

**해석**: topic 역참조로 4개가 guaranteed, 추가로 텍스트 매칭으로 10개 더 포함. 최종 14개 소스 커버. 비교적 깔끔한 결과.

---

### Case C: 글로벌 키워드 — `"전체 요약"`

```
hasGlobalKeyword: true
Tier 1 매칭: 없음
→ Tier 0 적용: 4개 에이전트 모두 호출, isGlobal=true

각 에이전트 chunkCap = allowedSources.length (제한 없음)
```

**해석**: 모든 위키의 모든 소스 첫 청크가 LLM에게 전달됨. 토큰 비용은 크지만 누락 없음. **단, 깊이는 얕음** (소스당 1청크).

---

### Case D: 무관 쿼리 — `"오늘 날씨"`

```
Router Tier 1: 없음
Router Tier 2: senate 등 모든 에이전트의 preScore 호출
  → "날씨"가 어떤 metadata와도 매칭 안 됨
Last-resort: 모두 호출

[senate agent]
  matchedEntities/Topics: (없음)
  guaranteedIds: 0
  chunkCap: 15
  totalScored: 1개 청크 (단어 "오늘"이 어디 한 곳에 등장)
  finalSources: ['18기-16차']
  confidence: 0.8 ← 1개 청크라도 매칭되면 0.8
```

**문제점**: 단 1개의 우연한 매칭으로 confidence가 0.8이 됨. confidence 임계값 0.3을 통과하므로 LLM에 의미 없는 컨텍스트가 전달됨. **개선 여지**: confidence 계산을 청크 점수의 절대값으로 변경.

---

### Case E: 모호한 쿼리 — `"최근 결정"`

```
Router Tier 1: 없음
Router Tier 2: 통과 (어느 위키도 "최근", "결정"이 metadata에 직접 매칭 안 됨)
Last-resort: 모두 호출

[senate agent]
  guaranteedIds: 0
  chunkCap: 15  ← entity 매칭 없으니 15로 떨어짐
  totalScored: 23개
  finalSources: 18기-1차, 19기-1차, 18기-18차, ...
  confidence: 0.8
```

**문제점**: "결정"이라는 매우 흔한 단어가 본문에 자주 등장 → 23개 청크 매칭 → 15개 잘림. **2026년 신규 회의록도 포함되지만 무작위에 가까운 선별**.

---

### Case F: 다중 위키 동시 호출 — `"26년 이사회랑 평의원회 회의 기록"` (실제 사례)

```
Router Tier 1: senate(2건), board(3건), vision(2건)
  → senate, board, vision 호출 (plan 누락)

[senate agent]
  matchedEntities: 평의원회 + 4개 위원회
  matchedTopics: 3개
  guaranteedIds: 18개
  chunkCap: 30
  finalSources (30개): 18기-4차, 18기-14차, ..., 19기-4차(rank 7), ...,
                       19기-5차(rank 22), 19기-6차서면심의(rank 28)
  confidence: 0.8

[board agent]
  guaranteedIds: 26개 (전부)
  chunkCap: 30, finalSources: 26개

[vision agent]
  Tier 1 매칭("이사회","평의원회") 후 점수 매김 → context 포함
```

**확인된 버그** (이미 패치):
- 19기-4차의 헤더 청크(점수 15, 회의일 2026-02-05 포함)가 본문 청크(점수 29, 전임교원 책임시수)보다 점수 낮아 LLM에게는 본문만 전달됨
- LLM이 "2026년 평의원회 기록 없음"으로 잘못 답변
- 패치: `## 제목 (id) | 회의일: 2026-02-05` 형태로 source block 헤더에 날짜 명시 → LLM이 점수 낮은 청크에서도 회의일 인식 가능

---

### Case G: tier2 사용자 + 민감 쿼리 — `"총장추천위 활동"`

```
[senate]
  isSensitiveAllowed: false (tier2)
  allowedSources: 51 - {sensitive: true 소스들} = 약 47개
    ← 19기-4차, 5차, 17기-22차 등 본문에 "총장추천위" 포함된 소스 제외
  matchedTopics: 평의원회 정책연구과제 (등)
  ...
```

**부작용**: 본문 단순 언급만으로 sensitive로 분류되는 광범위 마스킹. 정책 연구 결과 보고서가 검열 대상이 됨. **개선 여지**: frontmatter에 명시적 `sensitive: true` 플래그를 추가하고 본문 매칭은 제거.

## 4. LLM 답변 단계

### 4.1 System Prompt (lib/llm/prompts.ts)

핵심 5가지 원칙:

1. **P1. 할루시네이션 절대 금지** — 제공 자료에 없는 내용 생성 금지, 빈 항목은 "미확인" 등으로 채우지 말고 생략
2. **P2. 인라인 출처 표기 필수** — 모든 사실에 `[위키명] 문서ID` 형식
3. **P3. 테마별 구조화** — 교육/연구/재정/캠퍼스/거버넌스 등으로 묶기
4. **P4. 교차 확인** — 여러 위키에서 동일 내용 시 신뢰도 명시
5. **P5. 한계 인정** — 자료 범위 한계 명시

추가 길이 정책:
- 구체적 질문 (특정 인물·정책) → 상세 서술
- 광범위 질문 ("전체") → 요약 테이블 우선
- 비교 질문 → 표
- 단순 조회 → 결론 먼저

### 4.2 User Message

```
## 위키 자료

### [평의원회] 관련 자료
[entityBlocks] (entity 매칭 시)
---
[sourceBlocks 1] = ## 제목 (id) | 회의일: YYYY-MM-DD\n[chunk]
---
[sourceBlocks 2] = ...

---

### [이사회] 관련 자료
...

---

## 질문
[user query]
```

## 5. 확인된 동작상의 이슈와 권장 개선

| # | 이슈 | 위치 | 영향 | 상태 |
|---|---|---|---|---|
| 1 | 헤더 청크의 회의일이 LLM에게 전달 안 됨 | wiki-agent.ts:207 | LLM이 날짜 인식 실패 | ✅ 패치됨 |
| 2 | 단 1개 청크 매칭만으로 confidence 0.8 | wiki-agent.ts:226 | 무의미 컨텍스트 통과 | ⚠️ 미해결 |
| 3 | sensitive 판정이 본문 substring 매칭 | build-wiki-data.ts:229 | 광범위 검열 | ⚠️ 미해결 |
| 4 | guaranteed 소스가 너무 많이 끌려옴 | wiki-agent.ts:90~103 | 청크 30개 안에서 깊이 얕아짐 | ⚠️ 미해결 |
| 5 | "최근", "결정" 등 흔한 단어가 무작위 선별 유발 | wiki-agent.ts:33 | 노이즈 청크 | ⚠️ 미해결 |
| 6 | preScore의 양방향 includes가 너무 관대 | wiki-agent.ts:52 | Tier 2 필터 무력화 | ⚠️ 미해결 |
| 7 | 한국어 형태소 분석 없음 ("이사회랑"≠"이사회") | 전반 | 조사 붙은 단어 매칭 누락 | ⚠️ 미해결 |

### 5.1 우선순위별 권장사항

**우선순위 1 (정확도)**
- 이슈 #2: confidence를 `최고 청크 점수 / 쿼리 단어 수`로 정규화
- 이슈 #7: 조사 분리 (간단한 trailing-particle stripping 또는 nori-style tokenizer)

**우선순위 2 (검열 정확성)**
- 이슈 #3: Obsidian frontmatter에 명시 플래그 사용, 본문 매칭은 제거 또는 매우 좁힘

**우선순위 3 (응답 품질)**
- 이슈 #4: guaranteed 소스 수에 상한 (예: max 10개) 또는 점수 가중치 차등화
- 이슈 #5: 쿼리 단어 IDF 가중치 (전체 코퍼스에서 흔한 단어는 감점)

**우선순위 4 (장기)**
- 벡터 임베딩 검색 도입 (BGE-m3 등 다국어 모델)
- 텍스트 매칭 점수와 임베딩 유사도를 하이브리드로

## 6. 부록: 점수 계산 워크스루

`19기-4차` 소스에서 `26년 서울대 이사회랑 평의원회 회의 기록 알려줘` 쿼리 처리 예시:

**queryWords**: `["26년", "서울대", "이사회랑", "평의원회", "회의", "기록", "알려줘"]`

**Source-level pre-score**:
- guaranteedIds 포함 (entity `평의원회` 매칭) → +5
- topic `대학운영계획` 매칭 (`대학운영계획` 단어가 query에 없으므로) → 0
- tag `19기` (query에 없음) → 0
- tag `본회의` (query에 없음) → 0
- content에 `평의원회`, `회의` 등 등장 → +N
- **합계**: 약 7~10

**Chunk 점수** (5개 청크):

| 청크 | 본문 일부 | rawScore | guaranteed boost | 최종 |
|---|---|---|---|---|
| 0 | `# 제19기 평의원회 제4차 본회의 요약 / 회의일: 2026-02-05` | 7 | ×2+1 | **15** |
| 1 | `## 회의 성격 / 주요 결과 / 전임교원 책임시수...` | 14 | ×2+1 | **29** ← 선택 |
| 2 | `## 등장 인물` | 0 | (제외) | — |
| 3 | `## 연결 Topics` | 1 | ×2+1 | 3 |
| 4 | `## 특이사항` | 2 | ×2+1 | 5 |

**왜 청크 1이 청크 0보다 점수 높은가?**
- 청크 1 본문이 더 길어서 `평의원회`, `회의`, `19기` 등 키워드가 더 많이 등장
- 청크 0(헤더)는 짧아서 매칭 빈도가 낮음
- → **헤더가 점수에 의해 자연 도태됨** (이게 버그였음)

**패치 후**: source block 출력에 `| 회의일: 2026-02-05` 추가 → 청크 1만 LLM에게 가도 회의일은 헤더에서 인식 가능.

## 7. 검증 방법

각 케이스 검증을 위한 시뮬레이션 스크립트는 `scripts/build-wiki-data.ts`와 동일한 방식으로 데이터를 로드하여 router/agent 로직을 재현할 수 있다.

권장: `scripts/simulate-retrieval.ts`를 신규 작성하여 쿼리 → 최종 LLM 컨텍스트까지 출력하는 디버그 도구를 만들면 향후 회귀 검증에 유용.
